# African Guard Architecture

African Guard is structured as a cloud-native FinCrime operating system. The local implementation runs with simulated adapters by default, but the service boundaries, data contracts, and deployment surfaces are production-oriented.

## Services

| Service | Runtime | Responsibility |
| --- | --- | --- |
| `apps/web` | Next.js | Analyst, investigator, compliance, admin, and partner operating console. |
| `services/api` | NestJS | Fraud scoring API, case management, payment recall, collaboration intelligence, WebSocket stream, RBAC, audit, and rate limiting. |
| `services/ai` | Internal service | African Guard Risk Analysis Service. It is reachable only from the API on private infrastructure. |
| `postgres` | PostgreSQL + Prisma | Core event, alert, payment, case, institution, audit, and feedback records. |
| `redis` | Redis Streams/cache | Low-latency alert fanout, rate state, and hot decision cache. |
| `kafka` | Kafka | Durable event bus for financial events and fraud intelligence propagation. |
| `neo4j` | Graph database | Identity graph, mule network detection, fraud-ring linkage, and graph risk propagation. |

## Real-Time Flow

1. Browser traffic uses the African Guard web app and same-origin gateway paths only.
2. The NestJS API verifies identity, roles, tenant claims, schemas, and rate limits before orchestration.
3. Internal risk, database, cache, broker, and graph services are reached only server-side.
4. An event is posted to `POST /v1/events/score` or generated by the authenticated WebSocket simulator.
5. African Guard Decision Intelligence computes rule, anomaly, behavior, graph, consortium, and AML scores.
6. Critical and high-risk decisions become live alerts.
7. Critical decisions create cases automatically.
8. The payment, graph, AML, learning, and alert panels receive fresh `operating.picture` events over authenticated WebSockets.
9. Analyst labels are accepted by the learning loop and queued for retraining.

## Security Model

- Production RBAC is enforced through signed JWTs or secure server sessions with role, institution, and tenant claims.
- Local development header auth is allowed only when explicitly enabled outside production and is injected server-side by the local web gateway.
- Rate limiting is active globally through NestJS throttling.
- Consortium sharing uses HMAC pseudonymisation and AES-256-GCM encrypted envelopes.
- Audit events are created for case status changes, payment recalls, auto-case creation, and learning labels.
- Public health responses return only minimal status.
- Frontend responses are sanitised to avoid stack traces, provider payloads, private model identifiers, executable rule expressions, full SAR narratives, and unnecessary personal data.

## Agentic Operations Layer

The agentic layer is designed to reduce repetitive analyst work while preserving governance:

- Data Analyst Agent: answers merchant, transaction, session, and user-pattern questions.
- Graph Analyst Agent: maps shared devices, users, emails, IPs, merchants, and payment instruments.
- OSINT Search Agent: summarizes registry, domain, adverse media, and ownership findings for EDD.
- Rule Assistant Agent: converts plain-language hypotheses into review, hold, block, or step-up logic with backtest summaries.
- Dispute Filing Agent: assembles chargeback evidence and Visa CE 3.0 packages.
- Agentic AML Ops: clears low-risk queues, escalates high-risk activity, and keeps audit trails.

The current implementation uses deterministic demo data and in-memory agent runs. The intended production adapter boundary is a warehouse or lakehouse query layer plus governed external OSINT providers, card-network dispute APIs, and case-management persistence.
- Secrets are represented as vault references in configuration and `.env.example`.
